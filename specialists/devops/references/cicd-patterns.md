# CI/CD Pipeline Design Reference

> **Confidence:** All recommendations are `[HYPOTHESIS]` (untested best-practice) unless marked `[TESTED: date]` or `[REVISED: date]`. Check `../results/` for empirical outcomes before applying advice.

---

## Pipeline Stage Order

Stages must run in this order. Each stage gates the next. `[HYPOTHESIS]`

```
lint (30s) → unit tests (1–2m) → build (2–5m) → integration tests (5–15m) → deploy → smoke tests
```

| Stage | Purpose | Fail action |
|-------|---------|-------------|
| **Lint** | Style, type errors, obvious bugs | Block merge — fastest feedback |
| **Unit tests** | Business logic correctness | Block merge |
| **Build** | Artifact compilation, bundle generation | Block merge |
| **Integration tests** | DB, HTTP, service boundaries | Block merge |
| **Deploy to staging** | Validate artifact in real env | Block production deploy |
| **Smoke tests** | Confirm critical paths work post-deploy | Trigger rollback if fail |
| **Deploy to production** | Ship | — |
| **Production smoke tests** | Confirm production is healthy | Page on-call if fail |

**Never skip stages for "small" changes.** The probability of a catastrophic failure is independent of how small you think the change is. `[HYPOTHESIS]`

## Fail Fast Principles `[HYPOTHESIS]`

Put the fastest, cheapest checks first. Developers get feedback in seconds, not minutes, for the most common failure types.

```
lint         ~30s   — catch syntax/type errors immediately
unit tests   ~1–2m  — catch logic errors
build        ~2–5m  — catch compilation errors
integration  ~5–15m — catch integration errors (use cache to keep this fast)
```

**Rules:**
- If lint takes > 2 minutes, it is misconfigured or caching is broken — fix the pipeline, do not accept it
- If unit tests take > 5 minutes, investigate: tests may be hitting I/O (mocking is missing), or the suite needs parallelisation
- Integration tests that take > 20 minutes are often doing redundant work — audit and trim `[HYPOTHESIS]`
- Run stages in parallel where there is no dependency (e.g., lint and unit tests can run concurrently on separate runners)

## Caching Strategies

### npm / Node.js `[HYPOTHESIS]`

Cache `node_modules` keyed on the lockfile hash. If `package-lock.json` hasn't changed, restore the cache and skip install.

```yaml
# GitHub Actions example
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: ${{ runner.os }}-npm-
- run: npm ci
```

Use `npm ci` (not `npm install`) in CI — it installs from the lockfile exactly and is faster.

---

### Python / pip `[HYPOTHESIS]`

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements*.txt') }}
- run: pip install -r requirements.txt
```

Prefer `pip install --no-deps -r requirements.txt` with a locked requirements file generated by `pip-compile`. Avoid `pip install .` in CI without a lockfile — it resolves at install time and is non-deterministic. `[HYPOTHESIS]`

---

### Go modules `[HYPOTHESIS]`

```yaml
- uses: actions/cache@v4
  with:
    path: ~/go/pkg/mod
    key: ${{ runner.os }}-go-${{ hashFiles('**/go.sum') }}
- run: go mod download
```

---

### Docker layer caching `[HYPOTHESIS]`

Order Dockerfile instructions from least-to-most frequently changed:

```dockerfile
FROM node:20-alpine          # base (changes rarely)
WORKDIR /app
COPY package*.json ./         # manifests (change on dep changes)
RUN npm ci                   # install (cached if manifests unchanged)
COPY . .                     # source (changes every build)
RUN npm run build
```

Use `--cache-from` to reuse layers from the registry:

```bash
docker build \
  --cache-from registry.example.com/myapp:latest \
  --tag registry.example.com/myapp:$SHA \
  .
```

GitHub Actions: use `docker/build-push-action` with `cache-from: type=gha` for built-in layer caching.

## Secrets Handling

- [ ] **Never log secrets** — audit `echo`, `print`, `console.log` calls in pipeline scripts; mask secrets at the CI provider level
- [ ] **Use a secrets manager** — GitHub Actions Secrets, GitLab CI Variables (masked), HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager
- [ ] **Least privilege** — each job gets only the secrets it needs; do not pass all secrets to all jobs
- [ ] **Rotate on suspected compromise** — if a secret may have been logged or leaked, rotate it before investigating; do not wait for confirmation
- [ ] **Never hardcode** — no secrets in `.env` files committed to the repo, no secrets in Docker build args that end up in layers, no secrets in artifact metadata

**GitHub Actions secret masking:**
```yaml
- run: echo "::add-mask::${{ secrets.MY_SECRET }}"  # masks value in all subsequent log lines
```

**Vault dynamic secrets pattern (preferred for DB credentials):** `[HYPOTHESIS]`
- Pipeline requests a short-lived DB credential from Vault at job start
- Credential expires automatically after the job duration
- No long-lived DB password ever stored in CI

## Deployment Strategies

### Blue/Green `[HYPOTHESIS]`

Two identical production environments. Traffic switches from "blue" (current) to "green" (new) atomically.

| When to use | When NOT to use |
|-------------|----------------|
| Zero-downtime deploys required | Cost of running two full environments is prohibitive |
| Easy instant rollback (flip DNS/LB back) | State (DB, queue) is hard to share between environments |
| Stateless services | |

**Rollback:** switch load balancer target back to blue. Takes seconds.

---

### Canary `[HYPOTHESIS]`

Route a small percentage of traffic (e.g., 5%) to the new version. Monitor. Gradually increase to 100% if healthy, or roll back to 0%.

| When to use | When NOT to use |
|-------------|----------------|
| High-risk changes where you want real traffic validation | Cannot tolerate any users seeing the new version |
| Large user base where even 1% is meaningful signal | Breaking API change affects all users equally |
| Good observability to detect problems at 5% | |

**Percentages:** 5% → 25% → 50% → 100%, with monitoring pauses between each step. `[HYPOTHESIS]`

---

### Rolling `[HYPOTHESIS]`

Replace instances one-at-a-time (or in small batches), keeping the service available throughout.

| When to use | When NOT to use |
|-------------|----------------|
| Simple, low-risk updates | Breaking DB schema change (old + new code run concurrently) |
| K8s deployments (default behaviour) | Strict zero-downtime requirement (brief mixed-version state exists) |
| Cheap compared to blue/green | |

**DB migration rule with rolling deploys:** migrations must be backward-compatible with the old application version. Deploy in two steps: (1) migration + old code, (2) new code. `[HYPOTHESIS]`

## Health Check Pattern After Deploy `[HYPOTHESIS]`

Always run a health check after deployment before marking the deploy as successful or routing production traffic.

**Standard pattern:**
- 3 retries
- 10 seconds between retries
- Fail the pipeline (and trigger rollback) if all retries fail
- Check both `/healthz` (liveness) and a lightweight business logic probe

```bash
#!/bin/bash
HEALTH_URL="https://app.example.com/healthz"
MAX_RETRIES=3
RETRY_INTERVAL=10

for i in $(seq 1 $MAX_RETRIES); do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$HEALTH_URL")
  if [ "$STATUS" = "200" ]; then
    echo "Health check passed"
    exit 0
  fi
  echo "Attempt $i failed (HTTP $STATUS), retrying in ${RETRY_INTERVAL}s..."
  sleep $RETRY_INTERVAL
done

echo "Health check failed after $MAX_RETRIES attempts — triggering rollback"
exit 1
```

**What to check in the health endpoint:**
- Application process is running and accepting connections
- Database connectivity (lightweight query, not a full scan)
- Cache connectivity if cache is critical path
- Do NOT include external third-party API checks (you do not control their uptime)

## Key Syntax Differences: GitHub Actions vs GitLab CI vs CircleCI

| Concept | GitHub Actions | GitLab CI | CircleCI |
|---------|---------------|-----------|----------|
| Config file | `.github/workflows/*.yml` | `.gitlab-ci.yml` | `.circleci/config.yml` |
| Job definition | `jobs.<id>` | `<job-name>:` (top-level) | `jobs.<id>` |
| Triggers | `on: [push, pull_request]` | `rules:` or `only:/except:` | `triggers:` / `filters:` |
| Reusable steps | `uses: actions/<name>` | `include:` template | `orbs:` |
| Secrets access | `${{ secrets.NAME }}` | `$NAME` (auto-injected) | `$NAME` (set in project) |
| Cache | `actions/cache@v4` | `cache:` key/paths | `restore_cache` / `save_cache` |
| Artifacts | `actions/upload-artifact` | `artifacts: paths:` | `store_artifacts` |
| Matrix builds | `strategy.matrix` | `parallel:` or matrix | `matrix` (via parameters) |
| Service containers | `services:` block | `services:` block | `docker:` executor list |
| Stage ordering | Implicit via `needs:` | `stages:` array | `workflows:` requires |

**GitHub Actions minimal example:**
```yaml
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm test
```

**GitLab CI minimal example:**
```yaml
stages: [lint, test, build]
test:
  stage: test
  image: node:20
  cache:
    key: $CI_COMMIT_REF_SLUG
    paths: [node_modules/]
  script:
    - npm ci
    - npm test
```

**CircleCI minimal example:**
```yaml
version: 2.1
jobs:
  test:
    docker:
      - image: cimg/node:20.0
    steps:
      - checkout
      - restore_cache:
          keys: [v1-deps-{{ checksum "package-lock.json" }}]
      - run: npm ci
      - save_cache:
          key: v1-deps-{{ checksum "package-lock.json" }}
          paths: [~/.npm]
      - run: npm test
workflows:
  main:
    jobs: [test]
```

## Quick-Reference: Pipeline Smell → Fix

| Smell | Fix |
|-------|-----|
| Lint runs after unit tests | Move lint first — it's fastest |
| No cache on install step | Add cache keyed on lockfile hash |
| Secrets in plain `echo` or logs | Mask at CI level; audit log output |
| No health check after deploy | Add 3-retry health check before marking success |
| Integration tests take > 20 min | Parallelise, trim redundant tests, use faster DB fixtures |
| Rolling deploy breaks DB schema | Split into two deploys: migration first, code second |
| All jobs run sequentially | Add `needs:` / `depends_on` only where necessary; parallelise the rest |
